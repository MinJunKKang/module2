# 🔐 침해사고 재현 및 방어 실습 프로젝트

> 2025년 SKT 침해사고 공격 흐름을 실습 환경에서 재현하고,  
> 방어 조치 적용 전후를 직접 비교 실험하는 보안 분석 프로젝트

---

## 📌 프로젝트 개요

2025년 SKT 침해사고는 **웹쉘 미탐지, 평문 계정정보 노출, 암호화 미흡**이라는  
기본적인 보안 실수가 겹쳐 **2,696만 건의 유심정보**가 유출된 사건입니다.

본 프로젝트는 해당 공격 흐름을 AWS 환경에서 직접 재현하고,  
SKT가 권고받은 방어 조치를 적용했을 때 실제로 공격이 차단되는지 실험합니다.

> 💡 기본 보안 수칙만 지켰어도 막을 수 있었던 사고

---

## 🏗️ 시스템 아키텍처

    [공격자 - 개인 PC 브라우저]
            ↓ ① HTTP 접속 + 웹쉘 업로드
            ↓ ② 웹쉘 실행 → config.txt 읽기
    [서버A - AWS EC2]
      └ DVWA (취약한 웹 애플리케이션)
      └ config.txt (서버B IP / ID / PW 평문 저장)
            ↓ ③ SSH Lateral Movement (dbadmin@서버B_IP)
    [서버B - AWS EC2]
      └ MySQL (유심 더미 데이터)
            ↓
    [💀 유심 데이터 탈취 완료]

---

## ⚔️ 공격 시나리오 (1일차)

| 단계 | 내용 |
|------|------|
| 1단계 | 웹쉘 업로드 → 서버A 원격 명령 실행 권한 획득 |
| 2단계 | config.txt 탈취 → 서버B 접속 정보(ID/PW) 획득 |
| 3단계 | SSH Lateral Movement → 서버A에서 서버B로 횡적 이동 |
| 4단계 | MySQL 유심 DB 조회 및 mysqldump로 데이터 탈취 |

---

## 🛡️ 방어 조치 (2일차)

| 방어 | 기술 | 효과 |
|------|------|------|
| 방어1 | openssl AES-256-CBC로 config.txt 암호화 | 웹쉘로 읽어도 계정정보 획득 불가 |
| 방어2 | **GPT API 기반 LLM 웹쉘 탐지** | 일반/난독화/정상 PHP 3케이스 탐지, ClamAV 대비 성능 비교 |
| 방어3 | MySQL AES_ENCRYPT로 Ki값 암호화 | 탈취해도 암호화된 값만 노출 |

---

## 🤖 LLM 웹쉘 탐지 (핵심 기능)

기존 ClamAV는 알려진 패턴만 탐지 가능 → **난독화 웹쉘 우회 가능**  
GPT API는 코드의 의미를 이해하여 탐지 → **난독화 웹쉘도 탐지 가능**

    업로드된 PHP 파일
            ↓
    GPT API 전달
            ↓
    웹쉘 여부 판단 + 위험 이유 + 위험도 출력

**실험 케이스**
- ✅ 일반 웹쉘 (`<?php system($_GET['cmd']); ?>`)
- ✅ 난독화 웹쉘 (base64 인코딩 등)
- ✅ 정상 PHP 파일 (오탐 여부 확인)
